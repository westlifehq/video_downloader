# Workspace Rules (长期协作规约)

1. **响应语言与风格**：
   - 必须永远使用简体中文回复，遵循直接、专业、极简的沟通风格，不要废话，不要讲“Here's how you can...”。
   - 提供代码时提供精准的上下文 Diff，不要全盘输出未修改的长代码。
   - 遵从用户的 Prettier 格式化偏好。

2. **技术栈与开发约束**：
   - 尽量保持无第三方框架前端（Vanilla JS/CSS），降低复杂系统引入。修改 CSS 时继续延用已设定的变量（`--c-primary` 等）。
   - 后端新增依赖时务必三思，考虑对 `pkg` 打包或者今后 Docker 化是否产生阻碍（如原生二进制库 `better-sqlite3` 等应避免或小心测试）。

3. **系统与路径处理**：
   - 后端逻辑必须兼顾 Mac、Windows、Linux（NAS环境）。凡涉及文件路径的操作，一律强制使用 `path.join` 避免斜杠方向硬编码引发的错误。
   - 凡涉及执行系统命令（`child_process`），必须做不同 `process.platform` 的针对性判断。

4. **安全与异常处理**：
   - 本地文件的 `delete` 操作强制前置用户确认，且后端执行必须包含完整的路径合法性校验与 `try-catch` 容错处理。
   - 不要轻易暴露出跨域限制或随意更改 `CORS`，除非作为 API 明确拆分。

5. **用户意图预期**：
   - 把用户当成是熟练的技术决策者。对于新特性需求，优先分析商业与实现可行性（比如放弃不靠谱的小程序），主推最高效可靠的方案。

6. **环境与部署隔离**：
   - Docker / NAS 环境专用的变动需要放入 `nas-deployment` 等独立目录中隔离打包，必须保障根目录直接供本地 PC (`npm run dev`) 或单体打包 (`pkg`) 使用时不受污染或产生未预期的依赖。
