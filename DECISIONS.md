# 核心决策 (Decisions)

## 1. 放弃小程序/App，采用 Web + Docker (NAS部署)
**背景**：用户期望在移动端（安卓/微信）使用此下载服务。
**决策**：放弃开发原生 App 或微信小程序，转向“Web PWA + 飞牛 OS NAS Docker 部署”方案。
**原因**：
- 微信小程序对于“无水印下载”、“外部视频抓取”类目有极其严格的审核机制，几乎无法通过个人身份上架，且需要配置 HTTPS 和备案域名，合规成本极高。
- 原生 App 需要重新搭建跨平台工程（Flutter/UniApp），且手机直连下载吃手机流量和存储，而非局域网/NAS大本营。
- 采用 Web 形式署在已有的 x86 飞牛 OS 服务器上，手机不仅可以通过浏览器 (PWA) 获得类似 App 的体验，而且实际视频文件下到了服务器(NAS)大容量硬盘内，便于直接供家里的媒体中心(Plex/Jellyfin)使用。

## 2. 交互逻辑：“打开”按钮行为变更为“定位文件”
**背景**：历史区原设计是点击文件夹图标打开视频直接播放。
**决策**：将其修改为通过文件管理器打开文件所在目录并自动选中该文件（跨 Windows / macOS / Linux 适配）。
**原因**：作为“下载器”，用户的核心目的是资产归档和二次剪辑，直接系统级定位到文件更符合重度剪辑用户的素材管理直觉。

## 3. 依赖库版本锁定 (axios)
**背景**：需要用 `pkg` 进行跨平台二进制打包。
**决策**：锁定 `axios` 版本为 `0.27.2`。
**原因**：更高版本的 axios 或动态依赖在经过 `pkg` 编译打入虚拟文件系统时会出现模块找不到的运行时错误，老版本更为稳定。

## 4. Docker 版本物理目录隔离
**背景**：需要将后端放到 NAS 的 Docker 内运行，涉及到绑定 `0.0.0.0`、剥离本地资源管理器相关的逻辑。
**决策**：不采用分支管理，而是在代码库同级开辟独立的 `nas-deployment` 目录，将变更部分隔离放置。
**原因**：- 保障用户主工作区和 PC 打包工具链绝对纯净，互不影响，维护起来更符合直觉。

## 6. 外网访问：飞牛原生 Docker 跑 DDNSTO
**背景**：原计划在 iStoreOS 虚拟机中运行 DDNSTO 中转。
**决策**：改为直接在飞牛 (FN OS) 的 Docker 容器中运行 DDNSTO，并配置 `DEVICE_NAME`。
**原因**：
- **资源效率**：相比运行整个虚拟机 + iStoreOS，单个容器极度节省 CPU 和内存资源。
- **稳定性**：减少了网络转换层级（虚拟机网桥），降低了延迟。
- **管理便捷**：直接在飞牛端管理，配置 Host 网络模式后，域名映射设置更直观。

## 5. 应对特殊移动端浏览器的 HTTP 下载策略
**背景**：夸克等国产手机浏览器在下载 HTTP 流文件时，经常会忽略标准的 `Content-Disposition` 导致文件名丢失或保存为 `.vdat` 格式。
**决策**：对于 `/api/file` 下载接口，强制响应头为 `application/octet-stream`，并且将需要保存的文件名 urlencode 后显式追加到 URL 的 pathname 末尾（如 `/api/file/[id]/[name].mp4`）。
**原因**：这是让此类浏览器放弃猜测且强制以指定名称落地的最兼容解法。
