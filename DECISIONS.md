# 核心决策 (Decisions)

## 1. 放弃小程序/App，采用 Web + Docker (NAS部署)
**背景**：用户期望在移动端使用此下载服务。
**决策**：放弃 App/小程序，转向“Web PWA + 飞牛 OS NAS Docker 部署”方案。
**原因**：审核机制太严、合规成本极高。Web 部署到服务器，文件直接入库大本营，多设备访问极佳。

## 2. 交互逻辑：“打开”按钮行为变更为“定位文件”
**背景**：历史区原设计是点击文件夹图标打开视频直接播放。
**决策**：修改为系统级定位到文件所在目录并选中文件（跨 Windows / macOS / Linux 适配）。
**原因**：作为“下载器”，用户的核心目的是资产归档和二次剪辑，直接系统级定位更符合剪辑工作流。

## 3. 依赖库版本锁定 (axios)
**背景**：需要用 `pkg` 进行跨平台二进制打包。
**决策**：锁定 `axios` 版本为 `0.27.2`。
**原因**：更高版本的 axios 或动态依赖在经过 `pkg` 编译时会出现模块找不到的运行时错误，老版本更为稳定。

## 4. 采用 Playwright 拦截方案实现收藏同步
**背景**：抖音 API 签名（X-Bogus, MsToken）算法复杂且更新频繁。
**决策**：不进行算法逆向，而是使用 Playwright 驱动真实浏览器执行操作。
**原因**：鲁棒性极高，只要浏览器能看，同步逻辑就不会失效。借助浏览器的 `user_data` 本地存储登录态，安全且高效。

## 5. 收藏同步：UI 呈现分组式管理
**背景**：用户可能有成千上万个收藏，同步面板太乱。
**决策**：将列表拆分为「📥 未下载」与「✅ 已下载（折叠）」。
**原因**：保持面板视觉焦点在“待办事项”上，由于已下载已归档，折叠展示更合理。同时增加重新下载、打开文件夹、物理删除按钮。

## 6. Docker 版本物理目录隔离
**背景**：需要将后端放到 NAS 的 Docker 内运行，涉及到绑定 `0.0.0.0`、剥离本地资源管理器相关的逻辑。
**决策**：不采用分支管理，而是在代码库同级开辟独立的 `nas-deployment` 目录，将变更部分隔离放置。
**原因**：保障用户主工作区和 PC 打包工具链绝对纯净。

## 7. 外网访问：飞牛原生 Docker 跑 DDNSTO
**背景**：原计划在 iStoreOS 虚拟机中运行 DDNSTO 中转。
**决策**：改为直接在飞牛 (FN OS) 的 Docker 容器中运行 DDNSTO，并配置 `DEVICE_NAME`。
**原因**：极其节省资源、网络延迟低且配置简单。

## 8. 应对特殊移动端浏览器的 HTTP 下载策略
**背景**：国产手机浏览器下载 HTTP 流文件易丢失文件名。
**决策**：强制 `application/octet-stream`，并在 pathname 末尾追加 urlencode 的文件名。
**原因**：目前最通用的兼容方案。
